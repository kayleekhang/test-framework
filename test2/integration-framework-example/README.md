# Integration Framework Example

This repo is a mocked integration test framework for many products. The goal is
to keep tests requirement-focused while product libraries hide the mechanics of
backend APIs, Selenium UI flows, media tools, and packet capture.

## Layout

```text
framework/
  config/        environment config models and loader
  infra/         CloudStack and Ansible boundaries
  media/         GStreamer-style media send/receive helpers
  network/       packet capture and pcap assertion facade
  products/      product-specific backend/UI/verifier adapters
  ui/            browser/Selenium boundary
tests/
  conftest.py    pytest fixtures for environment, products, capture, browser
  products/      product-owned suites with requirement/capability markers
ansible/
  playbooks/     deployment/configuration entrypoints
  inventory/     generated inventory examples
```

## Flow

```text
pytest
  -> create or attach to an environment config
  -> provision CloudStack VMs from templates
  -> apply Ansible configuration
  -> build product objects from the generated config
  -> drive backend, UI, media, and packet-capture helpers
  -> assert requirement outcomes
```

For the full VM provisioning and configuration flow, see
[Infrastructure Flow](docs/infra-flow.md).

## Product Libraries

Each product gets small classes grouped by responsibility:

```text
framework/products/product1/
  backend.py  calls APIs, sends/receives media, triggers backend behavior
  ui.py       Selenium page actions for products with web UIs
  verify.py   product-specific assertions
  product.py  combines backend, UI, and verifier into one object
```

Product-owned tests should read like requirements:

```python
transmission = product.backend.send_udp_request(receiver_ip=receiver.config.ip, count=3)
packets = traffic_capture.collect_for_transmission(transmission)
product.verify.assert_udp_sent_to(packets, receiver.config.ip)
```

## Test Organization

The main test tree is product-first:

```text
tests/
  products/
    product1/
      test_suite_1.py
    product2/
      test_suite_1.py
      test_performance.py
```

This is useful when products have different UIs, setup paths, performance
profiles, media modes, or special validation. Pytest markers provide the
cross-cutting traceability:

```python
pytestmark = [
    pytest.mark.product("product1"),
    pytest.mark.suite("suite1"),
]

@pytest.mark.requirement("REQ-001")
@pytest.mark.capability("send_udp")
@pytest.mark.network
def test_product1_sends_udp(...):
    ...
```

With that shape, `tests/products/product1/test_suite_1.py` can satisfy many
requirements without scattering product-specific workflow across hundreds of
requirement files.

Generic cross-product tests are still useful when the behavior is truly common
for every product. Those can live in `tests/capabilities/` later, but they
should not replace product-owned suites when the product workflow matters.

## Running

This repo is safe and mocked, so it runs without real VMs:

```bash
python -m pytest
```

Run one requirement:

```bash
python -m pytest --requirement REQ-001
```

Run one product:

```bash
python -m pytest --product product1
```

Run one product suite:

```bash
python -m pytest --product product1 --suite suite1
```

Run one capability across matching tests:

```bash
python -m pytest --capability send_udp
```

Repeat tests that request the `iteration` fixture:

```bash
python -m pytest --requirement REQ-001 --iterations 25
```

Attach to a config generated by Ansible or another provisioner:

```bash
python -m pytest --product-config=path/to/environment.json
```

## Scaling Notes

Add new products under `framework/products/productN/` only when behavior differs.
If a product follows the default backend/UI behavior, it can use the base
`Product` class directly. Keep Wireshark/tshark parsing in `framework/network`,
GStreamer commands in `framework/media`, and Selenium/browser behavior in
`framework/ui` so 35 products do not duplicate infrastructure logic.

For real deployments, this repo should normally run on a test harness VM or CI
runner. That harness provisions or attaches to product VMs, runs Ansible,
executes pytest, drives UI/API/media tools, captures traffic, and stores logs,
pcaps, screenshots, and reports.

## Important TLS Note

A root CA private key is not enough to decrypt ordinary captured TLS traffic.
For passive decryption you usually need TLS session secrets, such as a browser
or client-generated `SSLKEYLOGFILE`. If you control the products, clean test
approaches include exporting TLS session keys from clients, terminating TLS at a
test proxy, capturing plaintext at the app boundary, or logging structured
protocol events from the products.
